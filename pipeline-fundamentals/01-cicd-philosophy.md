# CI/CD 철학 — 수동 배포가 실패하는 이유

---

## 🎯 핵심 질문

이 문서를 읽고 나면 다음 질문에 답할 수 있습니다.

- 수동 배포가 반복적으로 실패하는 구조적 이유는 무엇인가?
- Continuous Integration이 "항상 배포 가능한 상태"를 어떻게 만드는가?
- CI가 없을 때 왜 머지 지옥(Merge Hell)이 발생하는가?
- CD(Continuous Delivery)와 CD(Continuous Deployment)는 무엇이 다른가?
- 파이프라인이 없는 팀과 있는 팀의 배포 신뢰도는 왜 시간이 지날수록 벌어지는가?

---

## 🔍 왜 이 개념이 실무에서 중요한가

배포는 개발 팀에게 가장 스트레스받는 순간이다. 금요일 오후 배포를 피하는 문화, 배포 당일 모든 팀원이 대기하는 관행, 장애 시 원인을 찾는 데 수 시간이 걸리는 현실 — 이 모든 것은 수동 배포 프로세스가 본질적으로 불안정하다는 증거다.

CI/CD는 이 불안정성을 구조적으로 제거하려는 시도다. 단순히 "자동화 도구를 도입하는 것"이 아니라, **코드를 작성하는 순간부터 프로덕션에 도달하는 순간까지의 전 과정을 신뢰 가능하게 만드는 설계 철학**이다.

## 😱 흔한 실수 (Before — 원리를 모를 때의 접근)

```
상황: 2주에 한 번 대규모 배포를 진행하는 팀

배포 당일 시나리오:
  오후 2시: 개발자 A, B, C, D가 작성한 코드를 수동으로 병합
  오후 3시: 로컬에서 테스트 (각자 환경이 다름)
  오후 4시: 스테이징 서버에 수동 배포
  오후 5시: 스테이징에서 정체불명의 오류 발생
            "내 로컬에서는 됐는데요?"
  오후 6시: 원인 불명으로 배포 보류 결정
  다음 날: 원인 추적 — 개발자 B가 환경변수를 누락했음

흔한 수동 배포의 실패 패턴:
  1. 체크리스트를 보면서 수동으로 명령어 실행
     → 한 단계를 건너뛰거나 순서를 바꿈
  2. "이번엔 이 파일도 복사해야 해" — 구두로 전달된 절차
     → 문서화되지 않은 암묵적 지식
  3. 배포 실패 → 롤백 시도
     → "롤백 명령어가 뭐였지?" — 롤백 절차도 수동
  4. 배포한 버전이 어떤 커밋인지 추적 불가
     → 이전 버전으로 돌아가도 같은 문제 재현 가능성
```

## ✨ 올바른 접근 (After — 원리를 알고 난 설계/운영)

```
CI/CD를 도입한 팀의 배포 시나리오:

  개발자 A: 기능 브랜치에 push
    → 자동으로 단위 테스트 실행 (3분)
    → 테스트 실패 시 즉시 알림 → 머지 차단

  PR 생성:
    → 자동으로 통합 테스트 실행
    → 코드 품질 게이트 통과 확인
    → 리뷰어에게 "테스트 통과" 표시 자동 업데이트

  main 브랜치 머지:
    → 자동으로 스테이징 배포 파이프라인 실행
    → 이미지 빌드 → 테스트 → 스테이징 배포 → 스모크 테스트

  프로덕션 배포:
    → 버튼 하나 (또는 자동) → 동일한 이미지 프로덕션 적용
    → 실패 시 이전 이미지로 자동 롤백

핵심 변화:
  수동: 절차를 사람이 기억하고 실행
  자동: 절차가 코드(YAML)로 정의되고 항상 동일하게 실행
```

## 🔬 내부 동작 원리

### 수동 배포의 3대 구조적 실패 원인

**1. 휴먼 에러 (Human Error)**

사람은 반복 작업에서 실수를 피할 수 없다. 배포 체크리스트가 20단계라면, 19번은 성공하고 1번은 실수한다. 문제는 그 1번이 언제인지 알 수 없다는 것이다.

```bash
# 수동 배포 체크리스트 예시 (실제로 이런 문서가 존재한다)
□ 1. 서버에 SSH 접속
□ 2. 현재 실행 중인 버전 확인
□ 3. 새 JAR 파일을 /opt/app/에 복사
□ 4. 기존 프로세스 PID 확인
□ 5. kill -15 {PID}
□ 6. 새 프로세스 시작
□ 7. 로그 확인
□ 8. 헬스체크 URL 브라우저로 확인
□ ...19개 이후 생략
```

이 체크리스트의 근본 문제: 6번과 7번 사이에 30초를 기다려야 한다는 사실이 문서에 없을 수 있다. 또는 특정 환경에서만 필요한 8-1단계가 존재하는데 새로운 팀원은 모른다.

**2. 환경 불일치 (Environment Inconsistency)**

"내 로컬에서는 됐는데"는 환경 불일치의 신호다. 개발자 로컬, CI 서버, 스테이징, 프로덕션의 환경이 다르면 특정 환경에서만 재현되는 버그가 필연적으로 발생한다.

```
개발자 로컬  → Java 21, macOS, 특정 라이브러리 버전
CI 서버      → Java 17, Ubuntu, 다른 라이브러리 버전
스테이징     → Java 21, Ubuntu, 환경변수 일부 누락
프로덕션     → Java 21, Ubuntu, 실제 DB 접속 (스테이징과 다른 스키마)
```

Docker 컨테이너 이미지는 이 문제를 해결한다. 이미지 안에 Java 버전, 의존성, 설정이 모두 포함되어 있어 어느 환경에서 실행해도 동일한 결과를 보장한다.

**3. 롤백 불가능 (Irreversible Deployment)**

수동 배포에서 롤백은 또 다른 수동 작업이다. 배포 실패 시 패닉 상태에서 롤백 절차를 다시 수행해야 한다. 이전 버전의 파일이 어디 있는지, 어떤 설정이 필요한지 기억해야 한다.

CI/CD 파이프라인에서 롤백은 이전 이미지 태그로 배포하는 것이다. 이미지는 변경되지 않는 불변 아티팩트이기 때문에, 이전 상태로 돌아가는 것이 확실하게 보장된다.

### Continuous Integration이 "항상 배포 가능한 상태"를 만드는 원리

CI의 핵심 전제는 **코드베이스가 항상 배포 가능한 상태**여야 한다는 것이다. 이를 위한 메커니즘:

```
기존 방식 (Big Bang Merge):
  개발자 A: 2주 동안 기능 브랜치에서 독립 개발
  개발자 B: 2주 동안 다른 기능 브랜치에서 독립 개발
  배포 전날: main에 병합 시도
    → 충돌 100줄 발생
    → 양쪽 코드 의도를 파악하며 수동 해결
    → 해결 과정에서 버그 유입
    → "머지 지옥" 시작

CI 방식 (Continuous Integration):
  개발자 A: 매일 또는 기능 단위로 main에 통합
  개발자 B: 매일 또는 기능 단위로 main에 통합
    → 충돌이 작은 단위로 즉시 감지
    → 원인이 명확한 상태에서 해결
    → main은 항상 테스트가 통과된 상태 유지
```

**Feature Flag(피처 플래그)** 는 이 원칙을 극단적으로 밀고 나간 개념이다. 완성되지 않은 기능도 main에 통합하되, 플래그로 활성화를 제어한다.

```java
// Feature Flag 예시 — 코드는 main에 있지만 비활성화 상태
if (featureFlags.isEnabled("new-checkout-flow")) {
    return newCheckoutService.process(order);
} else {
    return legacyCheckoutService.process(order);
}
```

### CD — Continuous Delivery vs Continuous Deployment

두 용어는 자주 혼동된다.

```
Continuous Delivery (지속적 전달):
  파이프라인의 끝에 "배포 준비 완료" 상태를 만드는 것
  실제 프로덕션 배포는 사람이 버튼을 누름
  → 비즈니스적 배포 시점 제어가 필요할 때

Continuous Deployment (지속적 배포):
  모든 테스트를 통과하면 자동으로 프로덕션 배포
  사람의 개입 없음
  → 빠른 피드백 루프가 최우선일 때

시각적 비교:
  Delivery:   push → 테스트 → 빌드 → [스테이징 배포] → 🔴 수동 승인 → 프로덕션
  Deployment: push → 테스트 → 빌드 → [스테이징 배포] → 자동 → 프로덕션
```

두 방식 모두 "항상 배포 가능한 상태"를 전제로 한다. 차이는 최종 트리거가 사람인지 자동인지다.

## 💻 실전 실험

### 실험 1: 수동 배포와 CI/CD 파이프라인의 신뢰도 차이 시뮬레이션

```yaml
# .github/workflows/ci.yml — 가장 단순한 CI 파이프라인
name: Continuous Integration

on:
  push:
    branches: ["**"]         # 모든 브랜치 push 시
  pull_request:
    branches: [main]         # main 대상 PR 시

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - name: 코드 체크아웃
        uses: actions/checkout@v4

      - name: Java 21 설정
        uses: actions/setup-java@v4
        with:
          java-version: '21'
          distribution: 'temurin'
          cache: 'gradle'

      - name: 테스트 실행
        run: ./gradlew test

      - name: 테스트 결과 업로드
        uses: actions/upload-artifact@v4
        if: always()            # 실패해도 결과 업로드
        with:
          name: test-results
          path: build/reports/tests/
```

이 파이프라인이 하는 일:
- 모든 push마다 동일한 Java 버전, 동일한 명령어로 테스트 실행
- 사람이 체크리스트를 실행하지 않아도 됨
- 결과가 GitHub UI에 자동 표시

### 실험 2: 머지 차단 설정

```yaml
# .github/workflows/pr-check.yml
name: PR Quality Gate

on:
  pull_request:
    branches: [main]

jobs:
  quality-gate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-java@v4
        with:
          java-version: '21'
          distribution: 'temurin'
          cache: 'gradle'

      - name: 단위 테스트
        run: ./gradlew test

      - name: 커버리지 확인 (80% 미만 실패)
        run: ./gradlew jacocoTestCoverageVerification

      - name: 정적 분석
        run: ./gradlew checkstyleMain spotbugsMain
```

GitHub 레포지토리 설정에서 Branch Protection Rule → "Require status checks to pass before merging" 활성화하면 이 파이프라인이 통과해야만 PR 머지가 가능해진다.

## 📊 성능/비용 비교

| 항목 | 수동 배포 | CI/CD 파이프라인 |
|------|---------|----------------|
| 배포 소요 시간 | 1~3시간 (체크리스트 수행) | 10~20분 (자동화) |
| 배포 실패율 | 20~40% (경험 기반) | 5~10% (코드 기반) |
| 롤백 소요 시간 | 30분~2시간 | 3~5분 (이전 이미지 재배포) |
| 배포 빈도 | 주 1회 ~ 월 1회 | 하루 수십 회도 가능 |
| 장애 감지까지 시간 | 배포 후 수동 확인 | 자동 헬스체크 즉시 감지 |
| 신규 팀원 배포 가능 여부 | 교육 후 수 주 | 즉시 (YAML 이해만 필요) |

## ⚖️ 트레이드오프

**CI/CD 도입의 초기 비용**

파이프라인을 처음 구축하는 데는 시간이 걸린다. 테스트 코드가 없으면 CI의 효과가 반감된다. 테스트 코드 작성 → 파이프라인 구축 → 배포 자동화의 순서로 진행해야 하므로 초기 투자가 필요하다.

**자동화된 테스트의 한계**

CI가 통과해도 비즈니스 로직의 의도가 맞는지는 테스트가 판단하지 못한다. "항상 배포 가능한 상태"는 기술적 품질(컴파일 가능, 테스트 통과)을 보장하지, 비즈니스적 정확성을 보장하지 않는다.

**파이프라인 자체의 장애**

CI 서버가 죽으면 배포가 막힌다. 파이프라인 자체도 인프라이므로 가용성 관리가 필요하다. GitHub Actions는 GitHub의 가용성에 의존한다.

## 📌 핵심 정리

- 수동 배포는 **휴먼 에러**, **환경 불일치**, **롤백 불가능** 이라는 3대 구조적 문제를 가진다
- CI는 코드 통합을 지속적으로 수행해 **머지 지옥을 방지**하고 코드베이스를 항상 배포 가능한 상태로 유지한다
- CD(Delivery)는 배포 준비 자동화, CD(Deployment)는 배포 자체 자동화다
- 파이프라인의 본질은 **절차를 코드로 정의**해 동일한 결과를 항상 보장하는 것이다
- Docker 이미지는 환경 불일치를 해결하는 핵심 도구다 — 빌드된 이미지가 어느 환경에서나 동일하게 동작함을 보장한다

## 🤔 생각해볼 문제

**Q1.** 팀이 "코드를 main에 직접 push하고 즉시 배포"하는 방식을 유지하면서 "항상 배포 가능한 상태"를 보장할 수 있는가?

<details>
<summary>해설 보기</summary>

불가능하다. main에 직접 push하면 테스트가 실행되기 전에 코드가 통합된다. 테스트가 실패한 코드가 main에 들어가면 그 순간부터 main은 "항상 배포 가능한 상태"를 잃는다.

해결책은 두 가지다: (1) PR을 통해 테스트 통과를 확인 후 머지, (2) main에 직접 push하더라도 파이프라인이 실패하면 즉시 알림을 받고 핫픽스 commit으로 복구한다. 두 번째 방식은 일시적으로 main이 깨지는 것을 허용하지만, 빠른 복구가 전제다.

</details>

**Q2.** 테스트가 전혀 없는 레거시 프로젝트에 CI를 도입할 때 어디서부터 시작해야 하는가?

<details>
<summary>해설 보기</summary>

빌드 성공 여부부터 CI로 확인한다. 테스트가 없더라도 `./gradlew build` 또는 `mvn package`가 성공하는지 자동으로 확인하는 것만으로도 의미가 있다. 그다음 가장 크리티컬한 비즈니스 로직부터 테스트를 추가하면서 커버리지를 점진적으로 높인다. 완벽한 테스트를 기다리다가 CI를 도입하지 못하는 것보다, 불완전하더라도 먼저 파이프라인을 구축하는 것이 낫다.

</details>

**Q3.** Continuous Deployment(자동 프로덕션 배포)가 모든 팀에 적합한가?

<details>
<summary>해설 보기</summary>

아니다. 금융, 의료, 법률 관련 소프트웨어처럼 배포마다 규제 기관의 승인이 필요한 경우 Continuous Deployment는 적합하지 않다. 이런 경우 Continuous Delivery — 즉, 항상 배포 가능한 상태를 유지하되 최종 배포는 사람이 승인하는 방식 — 이 더 적합하다. 배포 빈도보다 배포 안전성과 규제 준수가 우선인 도메인을 파악하는 것이 중요하다.
